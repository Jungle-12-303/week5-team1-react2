# 집중 루틴 보드 서비스 기획서

## 1. 문서 목적

본 문서는 week5 v3 라이브러리 시연용 애플리케이션인 `집중 루틴 보드`를 실제 프로덕션 서비스처럼 설계하기 위한 기획 문서다.
기존 단일 페이지 데모 구조를 폐기하고, `대시보드 중심 + 기능 단위 다중 페이지` 구조로 재정의한다.
단, 여기서 말하는 다중 페이지는 여러 HTML 엔트리가 아니라 `단일 루트 엔트리에서 상태로 전환되는 다중 페이지 SPA`를 의미한다.

문서의 목표는 아래와 같다.

- 발표용 데모가 아니라 실제 서비스처럼 설계된 정보 구조를 만든다.
- v3 라이브러리의 핵심인 `Component + State + Hooks + Virtual DOM + Diff + Patch`를 여러 페이지에서도 일관되게 보여준다.
- 모바일과 데스크톱 모두에서 자연스러운 화면 구성을 갖춘다.
- 이후 구현 단계에서 화면 분해, 상태 구조, 테스트 전략까지 바로 이어질 수 있게 한다.

## 2. 제품 한 줄 정의

`집중 루틴 보드`는 사용자의 하루 집중 루틴을 계획하고, 실행하고, 되돌아보는 과정을 관리하는 개인 생산성 서비스다.

이 서비스는 단순한 할 일 목록이 아니라 아래 세 흐름을 한 제품 안에서 연결한다.

- 오늘 무엇에 집중해야 하는가
- 지금 어떤 루틴을 처리해야 하는가
- 오늘의 루틴 결과를 어떻게 해석하고 개선할 것인가

## 3. 제품 방향

### 3.1 핵심 방향

- 메인 페이지는 요약 중심의 `대시보드`로 유지한다.
- 세부 작업은 별도 기능 페이지로 분리한다.
- 사용자에게 한 화면에서 너무 많은 리스트와 조작 요소를 동시에 보여주지 않는다.
- 모바일에서는 페이지 분리가 기본 전략이다.
- 루트 상태 하나가 페이지 전환과 데이터 갱신을 모두 통제하도록 설계한다.

### 3.2 왜 다중 페이지가 맞는가

기존 단일 페이지 보드는 아래 문제를 가진다.

- 정보 밀도가 높아 한눈에 핵심을 파악하기 어렵다.
- `Focus Now / In Progress / Done` 3열 구조가 모바일에서 자연스럽지 않다.
- 추가, 필터, 정렬, 통계가 한 화면에 겹치며 조작 피로가 커진다.
- 대시보드와 작업 화면의 목적이 서로 섞인다.

따라서 서비스 구조는 `요약 화면`과 `작업 화면`을 분리해야 한다.

## 4. 최종 정보 구조

서비스는 아래 4개 페이지로 구성한다.

1. `대시보드`
2. `루틴 관리`
3. `인사이트`
4. `설정`

이 네 페이지는 데스크톱에서는 상단 내비게이션 또는 사이드 내비게이션으로 이동하고, 모바일에서는 하단 내비게이션으로 이동한다.
기술적으로는 `src/app/main.js`와 `index.html`을 기준으로 하나의 루트 앱만 mount 하고, `currentPage` 상태를 통해 네 화면을 전환한다.

## 5. 페이지 설계

### 5.1 대시보드

#### 목적

- 오늘의 상태를 가장 빠르게 파악하는 페이지
- 사용자가 앱에 들어왔을 때 가장 먼저 보는 메인 화면
- 작업을 많이 하기보다 `무엇이 중요한지`를 보여주는 페이지

#### 핵심 질문

- 오늘 완료율은 어떤가
- 지금 가장 먼저 해야 할 루틴은 무엇인가
- 오늘 아직 남은 루틴은 몇 개인가
- 최근에 어떤 행동이 일어났는가

#### 포함 요소

- 상단 환영 헤더
- 오늘 날짜와 요약 문구
- KPI 카드
  - 전체 루틴 수
  - 완료 수
  - 남은 수
  - 진행률
- `지금 바로 집중할 루틴` 카드 1~3개
- 최근 액션 타임라인 요약
- 오늘 인사이트 요약 카드
  - 가장 많은 카테고리
  - 가장 높은 우선순위 그룹
  - 현재 필터 없는 기준 핵심 통계

#### 제외 요소

- 긴 카드 리스트 전체
- 상세 필터 바
- 정렬 컨트롤 전체
- 대량 편집 기능

#### 사용자 행동

- `루틴 관리로 이동`
- `인사이트 보기`
- `가장 중요한 루틴 빠른 완료 처리`

#### 시연 포인트

- 루틴 하나를 완료하면 KPI 카드, 진행률 바, 최근 액션, 인사이트 요약이 동시에 갱신된다.
- 루트 상태 하나가 여러 요약 컴포넌트에 동시에 반영되는 장면을 보여주기 좋다.

### 5.2 루틴 관리

#### 목적

- 루틴을 생성, 탐색, 편집, 완료, 삭제하는 실제 작업 중심 페이지
- 사용자가 가장 오래 머무르는 메인 업무 공간

#### 핵심 질문

- 새 루틴을 어떻게 추가할까
- 원하는 루틴만 어떻게 빠르게 찾을까
- 우선순위와 상태를 어떻게 바꿀까
- 여러 루틴 중 무엇을 먼저 처리할까

#### 포함 요소

- 페이지 헤더
- 새 루틴 작성 패널
  - 제목 입력
  - 카테고리 선택
  - 우선순위 선택
  - 제출 버튼
- 필터 도구 모음
  - 상태 필터
  - 중요도 필터
  - 검색
  - 정렬
- 결과 요약 줄
  - 현재 보이는 개수
  - 적용 중인 필터 칩
- 루틴 리스트
  - 기본은 단일 컬럼 리스트
  - 데스크톱에서만 2열까지 허용
- 카드 액션
  - 완료/다시 진행
  - 삭제
  - 중요도/상태 시각 정보

#### 리스트 전략

- 기존 `3개 보드 병렬 열` 구조는 폐기한다.
- 기본 리스트는 필터와 정렬을 통해 하나의 흐름으로 본다.
- 상태 그룹은 필요한 경우 `섹션 헤더`로만 나누고, 병렬 보드는 사용하지 않는다.
- 데스크톱에서도 보드형 Kanban처럼 보이게 하지 않는다.

#### 사용자 행동

- 루틴 추가
- 검색
- 필터 변경
- 정렬 변경
- 완료 토글
- 삭제

#### 시연 포인트

- 입력과 리스트 조작이 가장 많이 일어나는 페이지라서 `input`, `change`, `submit`, `click` 이벤트를 집중적으로 보여줄 수 있다.
- `useMemo`의 파생 계산을 설명하기 가장 좋은 페이지다.

### 5.3 인사이트

#### 목적

- 단순 숫자 나열이 아니라, 오늘의 루틴 데이터를 해석하는 페이지
- 사용자가 실행 결과를 되돌아보는 회고 화면

#### 핵심 질문

- 어떤 카테고리에 시간을 가장 많이 쓰고 있는가
- 완료율은 어떤가
- 높은 우선순위 루틴을 얼마나 처리했는가
- 검색/필터 없이 전체 기준으로 오늘 상태는 어떤가

#### 포함 요소

- 인사이트 요약 헤더
- 완료율 카드
- 카테고리 분포 카드
- 우선순위별 분포 카드
- 최근 액션 로그
- 오늘의 추천 메시지
  - 예: `코딩 루틴 비중이 높습니다`
  - 예: `높음 우선순위 루틴이 아직 2개 남아 있습니다`

#### 시각화 방향

- 복잡한 차트 라이브러리는 쓰지 않는다.
- 카드, 막대형 요약, 분포 리스트, 비율 텍스트로 표현한다.
- v3 범위 안에서 구현 가능한 정적-반응형 시각 요소로 제한한다.

#### 사용자 행동

- 오늘 상태 확인
- 최근 액션 검토
- 필요한 경우 루틴 관리 페이지로 이동

#### 시연 포인트

- 같은 원본 상태가 다른 형태의 파생 데이터로 표현되는 장면을 보여준다.
- `useMemo`가 왜 필요한지 대시보드보다 더 직접적으로 보여줄 수 있다.

### 5.4 설정

#### 목적

- 서비스 자체 동작을 조절하는 보조 페이지
- 발표용으로는 비중이 낮지만, 실제 서비스처럼 보이게 하는 마무리 페이지

#### 포함 요소

- 기본 정렬 설정
- 기본 필터 초기값
- 로컬 저장 사용 여부 안내
- 테스트/런타임 안내 텍스트

#### 구현 우선순위

- 1차 구현에서는 간단한 정보 페이지 수준으로 시작 가능
- 상태 기반 설정 저장은 선택

## 6. 데스크톱 UX 전략

### 6.1 내비게이션

- 좌측 사이드 네비게이션 또는 상단 탭 사용
- 페이지 전환은 라우터 없이 루트 상태 `currentPage`로 처리

### 6.2 레이아웃

- 대시보드: 카드 그리드 기반
- 루틴 관리: 좌측 작성 패널 + 우측 리스트 또는 상단 작성 패널 + 하단 리스트
- 인사이트: KPI 카드 + 분포 영역

### 6.3 시각 스타일

- 카드 기반 레이아웃 유지
- 정보의 위계를 분명히 한다.
- 요약 카드와 작업 패널은 서로 다른 시각 밀도를 가져야 한다.
- 리스트는 세로 흐름을 기본으로 하되, 카드 디자인으로 지루하지 않게 만든다.

## 8. 루트 상태 구조

모든 상태는 루트 `App`에 둔다.

권장 상태 구조 예시는 아래와 같다.

```js
const [currentPage, setCurrentPage] = useState("dashboard");
const [routines, setRoutines] = useState([...]);
const [draftTitle, setDraftTitle] = useState("");
const [draftCategory, setDraftCategory] = useState("coding");
const [draftPriority, setDraftPriority] = useState("medium");
const [statusFilter, setStatusFilter] = useState("all");
const [priorityFilter, setPriorityFilter] = useState("all");
const [searchKeyword, setSearchKeyword] = useState("");
const [sortMode, setSortMode] = useState("created");
const [lastAction, setLastAction] = useState("오늘의 집중 루틴을 준비했습니다.");
const [settings, setSettings] = useState({
  defaultSortMode: "created",
  defaultPage: "dashboard",
  animationEnabled: true,
});
```

### 8.1 상태 역할

- `currentPage`
  - 현재 활성 페이지
- `routines`
  - 전체 루틴 원본 데이터
- `draftTitle`, `draftCategory`, `draftPriority`
  - 루틴 작성 입력값
- `statusFilter`, `priorityFilter`, `searchKeyword`, `sortMode`
  - 루틴 관리 페이지의 작업 상태
- `lastAction`
  - 최근 액션 메시지
- `settings`
  - 설정 페이지에서 관리하는 서비스 보조 상태

### 8.2 루틴 데이터 모델

루틴 항목은 최소한 아래 필드를 가진다.

```js
{
  id: "routine-1",
  title: "알고리즘 문제 2개 풀기",
  category: "coding",
  priority: "high",
  done: false,
  createdAt: 1710000000000,
  updatedAt: 1710000000000,
}
```

필드 정의:

- `id`: 리스트 렌더링과 patch 식별에 사용하는 고유 키
- `title`: 사용자에게 보이는 루틴 제목
- `category`: `study`, `coding`, `health`, `life` 중 하나
- `priority`: `high`, `medium`, `low` 중 하나
- `done`: 완료 여부
- `createdAt`: 최초 생성 시각
- `updatedAt`: 마지막 수정 시각

선택 확장 필드:

- `note`
- `estimatedMinutes`
- `completedAt`

## 9. 파생 데이터 구조

`useMemo`는 아래 파생 데이터에 사용한다.

- `doneCount`
- `activeCount`
- `progressPercent`
- `visibleRoutines`
- `topPriorityRoutines`
- `categorySummary`
- `prioritySummary`
- `dashboardHighlights`
- `recentActionSummary`

예시:

```js
const visibleRoutines = useMemo(() => {
  return sortRoutines(
    filterRoutines(routines, {
      statusFilter,
      priorityFilter,
      searchKeyword,
    }),
    sortMode
  );
}, [routines, statusFilter, priorityFilter, searchKeyword, sortMode]);
```

## 11. 페이지 전환 설계

라우터를 도입하지 않고, 루트 상태 기반 페이지 전환으로 처리한다.

```js
const [currentPage, setCurrentPage] = useState("dashboard");
```

전환 규칙은 아래와 같다.

- 상단/하단 내비게이션 클릭 시 `setCurrentPage()` 호출
- 현재 페이지에 따라 해당 페이지 컴포넌트만 렌더
- 공통 상태는 유지
- 페이지 이동 후에도 입력값, 필터, 루틴 데이터는 보존

이 구조는 v3 라이브러리 제약과 잘 맞는다.

### 11.1 내비게이션 계약

- 데스크톱은 상단 탭 또는 좌측 네비게이션을 사용한다.
- 모바일은 하단 4탭 내비게이션을 사용한다.
- 탭 순서는 `대시보드 → 루틴 → 인사이트 → 설정`으로 고정한다.
- 현재 활성 탭은 색상과 라벨 강조로 구분한다.
- 페이지 전환 시 스크롤 위치는 페이지 최상단으로 이동하는 것을 권장한다.

### 11.2 실제 기술 구조

- 이 서비스는 MPA가 아니다.
- `src/app/main.js`에서 하나의 `createApp()`만 mount 한다.
- `currentPage`가 바뀌면 같은 루트 앱이 다른 페이지 컴포넌트를 렌더한다.
- 따라서 v3 문서의 단일 루트 상태 구조와 충돌하지 않는다.

최소 수정 범위:

## 13. Hook 배치 전략

### 13.1 useState

- 페이지 전환 상태
- 루틴 원본 배열
- 입력 상태
- 필터 상태
- 마지막 액션

### 13.2 useEffect

- 현재 페이지와 완료율에 따라 문서 제목 갱신
- 루틴 원본을 localStorage에 저장

예시:

```js
useEffect(() => {
  document.title = `${pageTitleMap[currentPage]} · 집중 루틴`;

  return () => {
    document.title = "Week5 React-like Runtime";
  };
}, [currentPage, doneCount]);
```

### 13.3 useMemo

- 대시보드 KPI 계산
- 루틴 관리 필터 결과
- 인사이트 분포 계산
- 모바일/데스크톱 공통 요약 텍스트 계산

### 13.4 페이지별 상태 유지 정책

- `currentPage` 전환 시 `routines`는 항상 유지된다.
- 루틴 관리 페이지의 `searchKeyword`, `statusFilter`, `priorityFilter`, `sortMode`는 페이지를 벗어나도 유지하는 것을 기본 정책으로 한다.
- 작성 중인 `draftTitle`, `draftCategory`, `draftPriority`도 페이지 이동 후 유지할 수 있다.
- 설정 변경은 즉시 루트 상태에 반영되고, 이후 페이지 렌더에 공통 적용된다.
- 새로고침 시에는 localStorage에 저장된 루틴과 설정만 복원 대상으로 본다.

## 14. 시각 및 애니메이션 방향

### 14.1 원칙

- 애니메이션은 보조적이어야 한다.
- 정보 구조를 읽기 어렵게 만들면 안 된다.
- 페이지 전환과 카드 상태 변화만 가볍게 강조한다.

### 14.2 권장 애니메이션

- 페이지 전환 시 짧은 fade
- KPI 숫자/진행률 바 부드러운 갱신
- 루틴 카드 추가 시 짧은 slide-up
- 완료 처리 시 카드 opacity 변화
- 필터 칩 활성화 시 강조 색상 전환

### 14.3 금지

- 복잡한 모션 체인
- 긴 애니메이션
- 모바일에서 과도한 전환

## 15. 빈 상태와 오류 상태

실서비스 느낌을 위해 각 페이지는 빈 상태를 따로 가져야 한다.

### 15.1 대시보드 빈 상태

- 루틴이 없으면 `첫 루틴을 만들어 오늘 계획을 시작하세요` 메시지 노출

### 15.2 루틴 관리 빈 상태

- 검색 결과 없음
- 필터 결과 없음
- 전체 루틴 없음

이 세 가지를 구분한다.

### 15.3 인사이트 빈 상태

- 아직 분석할 루틴 데이터가 없다는 메시지 노출

### 15.4 오류 상태

- 저장 데이터 파싱 실패 시 기본 데이터로 fallback
- root 부재 시 명시적 오류

### 15.5 접근성 및 키보드 사용성

- 주요 버튼과 입력에는 명확한 텍스트 라벨이 있어야 한다.
- 키보드만으로 페이지 전환, 루틴 추가, 검색, 완료 처리, 삭제가 가능해야 한다.
- focus 이동 순서는 시각적 배치와 크게 어긋나면 안 된다.
- 모바일과 데스크톱 모두에서 버튼 클릭 영역은 충분히 커야 한다.

## 16. 테스트 전략

### 16.1 앱 기능 테스트

- 대시보드 렌더링
- 페이지 전환
- 루틴 추가 후 관리 페이지 즉시 반영
- 완료 토글 후 대시보드 KPI 반영
- 필터와 정렬 동작
- 삭제 후 인사이트/최근 액션 반영

### 16.2 모바일 관점 테스트

- 좁은 레이아웃에서도 핵심 블록이 세로 스택으로 정리되는지 확인
- 다중 열 보드가 다시 나타나지 않는지 확인

### 16.3 라이브러리 시연 테스트 포인트

- 페이지 전환도 결국 상태 변경이라는 점
- 페이지마다 다른 파생 데이터가 계산된다는 점
- 루트 상태 하나가 여러 페이지에서 일관되게 재사용된다는 점

### 16.4 페이지별 테스트 매트릭스

대시보드:

- KPI 카드 렌더링
- 빠른 완료 처리
- 최근 액션 반영

루틴 관리:

- 추가
- 수정
- 삭제
- 검색
- 상태/중요도 필터
- 정렬

인사이트:

- 완료율 계산
- 카테고리 분포
- 우선순위 분포
- 추천 메시지 갱신

설정:

- 기본 정렬 변경
- 기본 첫 화면 변경
- 애니메이션 on/off

## 17. 페이지별 승인 기준

### 17.1 대시보드 승인 기준

- KPI 카드가 루틴 상태와 즉시 동기화된다.
- 핵심 루틴 요약이 최소 1개 이상 노출된다.
- 최근 액션이 최신 상태를 반영한다.

### 17.2 루틴 관리 승인 기준

- 추가, 수정, 삭제, 토글, 검색, 필터, 정렬이 모두 동작한다.
- 새 루틴은 새로고침 없이 즉시 리스트에 반영된다.
- 필터 결과가 없을 때 빈 상태가 올바르게 보인다.

### 17.3 인사이트 승인 기준

- 카테고리 분포와 완료율이 루틴 데이터와 일치한다.
- 추천 메시지가 현재 상태를 설명한다.

### 17.4 설정 승인 기준

- 최소 2개 이상의 설정 항목이 존재한다.
- 설정 변경이 관련 화면에 즉시 반영된다.

## 19. 발표 시 강조 메시지

- 이 서비스는 단순 투두 리스트가 아니라 실제 서비스처럼 역할이 분리된 다중 페이지 구조를 가진다.
- 루트 상태는 하나지만, 대시보드, 관리, 인사이트가 서로 다른 형태로 그 상태를 보여준다.
- 페이지 전환, 필터, 정렬, 추가, 삭제, 완료 처리 모두 v3 라이브러리의 상태와 렌더 파이프라인 위에서 동작한다.
- `useEffect`는 문서 제목과 저장 효과를 처리하고, `useMemo`는 페이지별 파생 데이터 계산을 담당한다.

## 20. 최종 권장 요약

`집중 루틴 보드`는 이제 단일 페이지 데모가 아니라, 실제 프로덕션 서비스처럼 설계된 `다중 페이지 생산성 앱`으로 정의한다.

이 방향은 아래 이유로 적합하다.

- 대시보드가 진짜 대시보드처럼 보인다.
- 작업 페이지와 요약 페이지의 목적이 분리된다.
- 모바일 구조가 훨씬 자연스럽다.
- v3 라이브러리의 상태/렌더링 구조를 더 설득력 있게 보여줄 수 있다.

따라서 이후 구현은 본 문서를 기준으로 `대시보드 중심 다중 페이지 앱` 형태로 진행한다.
