# 카드 컬렉션 쇼케이스 서비스 기획서

## 1. 문서 목적

본 문서는 week5 v3 라이브러리 시연용 애플리케이션인 `카드 컬렉션 쇼케이스 서비스`를 설계하기 위한 기획 문서다.

이번 시연 앱은 다음 목적을 동시에 만족해야 한다.

- 청중이 보기에 즉시 흥미롭고 시각적으로 인상적이어야 한다.
- v3 라이브러리의 핵심인 `Component + State + Hooks + Virtual DOM + Diff + Patch`를 자연스럽게 드러내야 한다.
- 단순 카드 나열 데모가 아니라 실제 서비스처럼 정보 구조가 분리된 상태 기반 다중 페이지 SPA여야 한다.
- 외부 이미지를 활용하되, 화면 전환과 데이터 갱신은 v3 라이브러리가 담당해야 한다.

본 문서에서 말하는 다중 페이지는 여러 HTML 엔트리를 의미하지 않는다.
`src/app/main.js`와 `index.html`을 기준으로 하나의 루트 앱만 mount 하고, `currentPage` 상태를 통해 여러 화면처럼 보이게 전환하는 `상태 기반 다중 페이지 SPA`를 의미한다.

## 2. 제품 한 줄 정의

`카드 컬렉션 쇼케이스 서비스`는 외부 공개 API에서 받아온 몬스터 카드 이미지를 기반으로,
사용자가 카드를 탐색하고, 상세 정보를 보고, 즐겨찾기 컬렉션을 관리하고, 카드의 시각 효과를 체험하는 인터랙티브 갤러리 서비스다.

청중이 이 서비스를 보며 직관적으로 이해해야 할 메시지는 다음과 같다.

- 카드 목록이 필터와 정렬에 따라 즉시 바뀐다.
- 카드 선택 시 상세 패널과 통계가 함께 갱신된다.
- 즐겨찾기나 뷰 설정을 바꾸면 여러 페이지가 동시에 영향을 받는다.
- 카드의 3D 틸트와 하이라이트는 시각 효과이며, 실제 데이터 전환은 v3 렌더러가 담당한다.

## 3. 서비스 방향

### 3.1 핵심 방향

- 메인 페이지는 대시보드 역할만 수행한다.
- 실질적인 카드 탐색과 선택은 컬렉션 페이지에서 수행한다.
- 선택된 카드의 시각적 몰입감은 상세 페이지에서 극대화한다.
- 사용자의 취향과 저장 상태는 컬렉션, 상세, 설정 페이지에 걸쳐 공통으로 반영된다.
- 고빈도 마우스 이동 효과는 렌더러 전체 재렌더가 아니라 카드 단위 DOM 스타일 보조 효과로 처리한다.

### 3.2 왜 이 주제가 시연에 적합한가

이 주제는 다음 이유로 v3 라이브러리 시연에 적합하다.

- 카드 자체가 시각적으로 매력적이라 청중의 주목도가 높다.
- 카드 선택, 정렬, 필터, 즐겨찾기, 페이지 전환이 모두 명확한 상태 변화를 만든다.
- 외부 이미지 로딩이라는 실제 서비스 요소가 있어 데모의 설득력이 높다.
- 3D 틸트와 광택 효과로 “재미 요소”를 넣되, 상태 기반 렌더링과 역할을 분리해 설명할 수 있다.
- 모바일에서도 카드 중심 UI가 자연스럽다.

## 4. 서비스 정체성

서비스 톤은 다음 기준으로 잡는다.

- 단순 장난감이 아니라 실제 컬렉션 서비스처럼 보이게 한다.
- 포켓몬 카드 감성을 참고하되, 앱 이름과 UI는 특정 상표에 종속되지 않게 설계한다.
- 카드 이미지는 외부 공개 API나 공개 이미지 URL을 사용한다.
- 카드 프레임, 글로스, 포인터 반응은 직접 스타일링하여 구현한다.

권장 서비스 이름 예시:

- `Prism Dex`
- `Monster Card Atlas`
- `Holo Card Library`

## 5. 정보 구조

앱은 아래 4개 페이지로 구성한다.

1. `대시보드`
2. `컬렉션`
3. `카드 상세`
4. `설정`

데스크톱에서는 상단 내비게이션 또는 좌측 내비게이션으로 이동하고,
모바일에서는 하단 탭으로 이동한다.

기술적으로는 모두 하나의 루트 앱 안에서 `currentPage` 상태로 전환된다.

## 6. 페이지 설계

### 6.1 대시보드

#### 목적

- 오늘의 컬렉션 상태를 가장 빠르게 요약해 보여준다.
- 사용자가 어떤 카드를 먼저 보거나 탐색해야 하는지 안내한다.
- 서비스 전체를 한눈에 이해시키는 첫 화면 역할을 한다.

#### 사용자 질문

- 현재 로드된 카드 수는 몇 장인가
- 어떤 속성이나 타입의 카드가 많은가
- 즐겨찾기한 카드는 몇 장인가
- 지금 가장 주목할 카드가 무엇인가

#### 포함 요소

- 페이지 헤더
- 서비스 한 줄 소개
- KPI 카드
  - 전체 카드 수
  - 즐겨찾기 수
  - 현재 필터 기준 표시 카드 수
  - 선택된 카드 여부
- `오늘의 추천 카드` 혹은 `Spotlight Card`
- 최근 선택 카드 요약
- 타입 분포 또는 희귀도 분포 요약
- 컬렉션 페이지 이동 CTA

#### 제외 요소

- 긴 카드 그리드 전체
- 상세 스펙 전체
- 설정 제어 전체

#### 시연 포인트

- 카드 하나를 즐겨찾기하면 KPI, 추천 카드, 최근 액션이 함께 갱신된다.
- 루트 상태 하나가 여러 요약 컴포넌트를 동시에 바꾸는 장면을 보여주기 좋다.

### 6.2 컬렉션

#### 목적

- 실제 카드 탐색, 선택, 필터, 정렬을 담당하는 메인 작업 페이지다.
- 사용자가 가장 오래 머무는 중심 페이지다.

#### 사용자 질문

- 원하는 카드만 어떻게 빠르게 찾을까
- 어떤 기준으로 카드를 정렬할까
- 마음에 드는 카드를 어떻게 저장할까
- 어떤 카드를 상세 보기로 넘길까

#### 포함 요소

- 페이지 헤더
- 검색 입력
- 타입 필터
- 즐겨찾기 필터
- 정렬 선택
- 결과 수 요약
- 카드 그리드
- 카드 선택 상태 표시
- 카드 즐겨찾기 토글
- 카드 상세 페이지 이동

#### 카드 한 장의 포함 요소

- 카드 대표 이미지
- 카드 이름
- 카드 번호 또는 인덱스
- 타입 또는 속성 배지
- 즐겨찾기 버튼
- 기울기와 글로스 효과

#### 리스트 전략

- 기존 `3개 보드 병렬 열` 구조는 사용하지 않는다.
- 기본은 반응형 카드 그리드다.
- 데스크톱에서는 3~5열, 모바일에서는 2열 또는 1열을 사용한다.
- 검색과 필터 결과는 `useMemo` 파생 데이터로 계산한다.
- 전체 카탈로그가 커져도 전환 성능을 유지하기 위해 Collection은 가상 스크롤 기반 부분 렌더를 사용한다.
- 스크롤 높이는 전체 목록이 있는 것처럼 유지하되, 실제 DOM에는 현재 화면 근처 카드만 렌더한다.
- 데스크톱 Collection은 작업 중심 화면이므로, 메인 카드 영역이 최소 3열 이상 보이도록 화면 폭을 우선 배분한다.

#### 시연 포인트

- 검색어 입력 즉시 카드 수와 그리드가 변한다.
- 정렬 변경 시 카드 순서만 바뀌며 전체 앱을 새로 mount 하지 않는다.
- 카드 선택 시 상세 페이지 진입 없이도 선택 상태가 즉시 강조된다.
- 대량 카드 데이터도 현재 구간만 렌더된다는 점을 통해, 전체 새로고침이 아니라 필요한 영역만 갱신됨을 설명할 수 있다.

### 6.3 카드 상세

#### 목적

- 선택된 카드 한 장을 몰입감 있게 보여주는 페이지다.
- 3D 틸트와 조명 효과의 시각적 하이라이트를 담당한다.

#### 사용자 질문

- 이 카드가 왜 특별한가
- 큰 이미지로 보면 어떤가
- 어떤 타입과 속성을 가지는가
- 컬렉션 안에서 이 카드를 저장했는가

#### 포함 요소

- 큰 카드 프리뷰
- 보조 `Game Sprite` 비교 카드
- 카드 이름과 번호
- 타입, 키, 무게, 속성 등 요약 정보
- 즐겨찾기 버튼
- 컬렉션으로 돌아가기 버튼
- 이전/다음 카드 탐색 버튼
- 카드 배경 글로우 및 기울기 효과

#### 시각 전략

- 가장 화려한 화면이어야 한다.
- 카드 한 장이 화면 중심을 차지한다.
- 커서 이동에 따라 카드가 3차원으로 기울고, 하이라이트가 따라다닌다.
- 그러나 이 효과는 전체 루트 상태 재렌더가 아니라 카드 DOM 스타일에 국한된다.

#### 시연 포인트

- 카드 선택 상태가 상세 페이지의 큰 카드로 전이된다.
- 즐겨찾기 토글 시 대시보드와 컬렉션에 동시에 영향이 간다.
- 고빈도 효과와 상태 기반 렌더링을 분리한 설계를 설명하기 좋다.
- 고해상도 아트워크와 기본 스프라이트를 비교해, 같은 카드 데이터가 다른 DOM 속성 patch로 반영되는 장면을 보여주기 좋다.

### 6.4 설정

#### 목적

- 서비스 동작 방식과 시연 옵션을 제어하는 보조 페이지다.
- 발표 상황에서 데모 톤을 조정할 수 있는 안전한 조절 페이지다.

#### 포함 요소

- 기본 시작 페이지 선택
- 기본 정렬 기준 선택
- 카드 틸트 효과 on/off
- 광택 효과 on/off
- 이미지 고해상도 모드 on/off
- 샘플 데이터 초기화 또는 캐시 초기화

#### 시연 포인트

- 설정 변경이 즉시 다른 페이지의 표현 방식에 반영된다.
- 루트 상태가 앱 전역 설정을 통제한다는 점을 설명하기 쉽다.

## 7. 기술적 구현 원칙

### 7.1 v3 렌더러가 담당할 것

- 페이지 전환
- 외부 데이터 로드 후 목록 렌더
- 검색어 변경에 따른 결과 변화
- 정렬과 필터
- 카드 선택 상태
- 즐겨찾기 상태
- 설정 상태
- 상세 패널 정보 갱신
- 빈 상태와 오류 상태 전환

### 7.2 고빈도 시각 효과가 담당할 것

- 카드 기울기
- 카드 표면 글로스 이동
- 커서 위치 기반 반사광
- 카드 hover 확대감

이 효과는 가능한 한 다음 방식으로 처리한다.

- 카드 DOM element에 `mousemove` 이벤트 연결
- 마우스 좌표를 기반으로 CSS 변수 또는 inline style 업데이트
- 루트 `useState`로 매 프레임 렌더하지 않음
- 카드 포인터 이탈 시 원위치 복귀

이 원칙은 v3 라이브러리의 시연 포인트를 살리면서도 성능 부담을 줄이기 위한 것이다.

## 8. 외부 이미지 및 데이터 전략

### 8.1 데이터 소스 방향

권장 방향은 공개 API 기반 카드/몬스터 데이터를 사용하는 것이다.

후보:

- PokeAPI의 공개 포켓몬 데이터
- 공개 이미지 CDN
- 사전 정의한 카드 메타데이터 + 외부 이미지 URL 조합

### 8.2 데이터 사용 원칙

- 앱 시작 시 카드 목록 데이터를 로드한다.
- 필요한 최소 필드만 정규화하여 루트 상태에 저장한다.
- 이미지는 외부 URL을 `<img>`로 렌더한다.
- 이미지 로드 실패 시 fallback 이미지를 사용하거나 플레이스홀더 카드를 렌더한다.

### 8.3 카드 데이터 모델

권장 최소 모델:

```js
{
  id: "pokemon-025",
  name: "Pikachu",
  number: "025",
  imageUrl: "https://...",
  thumbUrl: "https://...",
  types: ["electric"],
  rarity: "rare",
  height: 0.4,
  weight: 6.0,
  baseStats: {
    hp: 35,
    attack: 55,
    defense: 40,
    specialAttack: 50,
    specialDefense: 50,
    speed: 90,
  },
  flavor: "...",
  isFavorite: false,
}
```

### 8.4 정규화 결과 저장 정책

- 외부 원본 응답 전체를 상태에 두지 않는다.
- 화면 렌더에 필요한 최소 필드만 추출한다.
- 이미지 URL과 메타데이터를 분리한다.

## 9. 루트 상태 구조

모든 상태는 루트 `App`에 둔다.

권장 상태 구조 예시:

```js
const [currentPage, setCurrentPage] = useState("dashboard");
const [cards, setCards] = useState([]);
const [isLoading, setIsLoading] = useState(true);
const [loadError, setLoadError] = useState(null);
const [selectedCardId, setSelectedCardId] = useState(null);
const [searchKeyword, setSearchKeyword] = useState("");
const [typeFilter, setTypeFilter] = useState("all");
const [favoritesOnly, setFavoritesOnly] = useState(false);
const [sortMode, setSortMode] = useState("number");
const [lastAction, setLastAction] = useState("카드 라이브러리를 불러오는 중입니다.");
const [settings, setSettings] = useState({
  defaultPage: "dashboard",
  defaultSortMode: "number",
  tiltEnabled: true,
  glareEnabled: true,
  highResImage: true,
});
```

### 9.1 상태 역할

- `currentPage`
  - 현재 활성 페이지
- `cards`
  - 외부에서 받아온 정규화 카드 목록
- `isLoading`
  - 초기 로딩 중 여부
- `loadError`
  - 외부 데이터 로드 실패 정보
- `selectedCardId`
  - 현재 선택한 카드
- `searchKeyword`, `typeFilter`, `favoritesOnly`, `sortMode`
  - 컬렉션 페이지 작업 상태
- `lastAction`
  - 공용 상태 메시지
- `settings`
  - 서비스 전역 표현 설정

### 9.2 상태 유지 정책

- 페이지를 이동해도 카드 목록은 유지된다.
- 검색어, 타입 필터, 정렬은 컬렉션 페이지를 벗어나도 유지한다.
- 선택된 카드도 상세 페이지와 대시보드 사이에서 유지한다.
- 설정은 localStorage에 저장해 새로고침 후 복원할 수 있다.

## 12. 페이지 전환 설계

라우터를 도입하지 않고, 루트 상태 기반 페이지 전환으로 처리한다.

```js
const [currentPage, setCurrentPage] = useState("dashboard");
```

전환 규칙은 다음과 같다.

- 상단/하단 내비게이션 클릭 시 `setCurrentPage()` 호출
- 카드 클릭 시 `selectedCardId`를 바꾸고 `currentPage = "detail"`로 이동
- `Back to Collection` 동작은 `currentPage = "collection"`으로 처리
- 모든 페이지는 같은 루트 앱 안에서 렌더된다

## 15. 외부 데이터 로딩 전략

### 15.1 초기 로드

- 앱 mount 후 `useEffect`에서 데이터 로드 시작
- 로딩 중에는 skeleton 또는 placeholder 카드 표시
- 성공 시 카드 목록 저장
- 실패 시 오류 상태 렌더
- 원격 카탈로그는 먼저 `10장 preview`를 빠르게 보여주고, 이후 전체 `#001 ~ #1025` 카탈로그를 백그라운드로 이어서 로드한다.
- 원격 로드 실패 시 마지막 성공 캐시가 있으면 캐시를 사용하고, 없으면 로컬 fallback 카드로 내려간다.

### 15.2 실패 처리

- 네트워크 오류 시 사용자에게 재시도 버튼 제공
- 일부 이미지 실패 시 카드 단위 fallback 렌더
- 전체 API 실패 시 앱 전체를 죽이지 않고 오류 페이지 또는 빈 상태 표시

### 15.3 저장 정책

- 즐겨찾기와 설정은 localStorage에 저장 가능
- 외부 원본 데이터 전체 캐시는 선택 사항
- week5 범위에서는 로딩 성공 후 메모리 상태 유지로 충분하다

## 16. 시각 및 애니메이션 전략

### 16.1 디자인 목표

- 단순한 카드 리스트가 아니라 프리미엄 컬렉션 서비스처럼 보여야 한다.
- 배경, 글로우, 하이라이트, 카드 프레임이 서비스의 개성을 만든다.
- 카드 상세 페이지는 가장 강한 인상을 남겨야 한다.

### 16.2 권장 시각 요소

- 깊이감 있는 배경 그라데이션
- 카드 타입별 배경색 포인트
- 유리질 패널 또는 홀로그램 질감
- 카드 hover 시 확대감
- 카드 표면 글로스 이동
- 페이지 전환 시 부드러운 fade/slide

### 16.3 카드 3D 틸트 규칙

- 커서 위치에 따라 `rotateX`, `rotateY`를 계산한다.
- 카드 표면에 radial gradient 하이라이트를 얹는다.
- 포인터 이탈 시 원위치로 부드럽게 복귀한다.
- 이 효과는 CSS 변수 또는 inline style로 처리한다.
- `mousemove`마다 루트 상태를 바꾸지 않는다.

### 16.4 설정과 연동할 것

- `tiltEnabled`
- `glareEnabled`
- `highResImage`

## 17. 모바일 전략

### 17.1 기본 원칙

- 카드 그리드는 모바일에서 정보 밀도를 줄여야 한다.
- 상세 정보는 한 화면에 다 넣지 않고 단계적으로 보여준다.
- 하단 탭 내비게이션을 사용한다.

### 17.2 모바일 페이지별 처리

- 대시보드
  - KPI 카드 2열
  - 추천 카드 1장 중심
- 컬렉션
  - 검색과 필터를 접을 수 있게 구성
  - 카드 2열 또는 1열
  - Inspector 패널은 숨겨서 카드 그리드가 우선 보이게 한다
- 상세
  - 큰 카드 1장 중심
  - 세부 정보는 아래 스택
- 설정
  - 스크롤형 단순 리스트

## 18. 접근성 및 사용성

- 카드 버튼과 즐겨찾기 버튼은 분리된 의미를 가져야 한다.
- 버튼은 명확한 텍스트나 aria-label을 가져야 한다.
- 키보드만으로 카드 탐색과 선택이 가능해야 한다.
- 포커스 스타일이 분명해야 한다.
- 이미지가 실패해도 카드 이름과 기본 정보는 읽을 수 있어야 한다.

## 19. 빈 상태와 오류 상태

### 19.1 대시보드 빈 상태

- 카드 로드 전이면 skeleton 카드 표시
- 카드가 비어 있으면 `컬렉션을 불러오지 못했습니다` 또는 `표시할 카드가 없습니다` 메시지 표시

### 19.2 컬렉션 빈 상태

- 검색 결과 없음
- 필터 결과 없음
- 즐겨찾기 결과 없음

이 세 가지를 구분한다.

### 19.3 상세 빈 상태

- 선택된 카드가 없으면 컬렉션 페이지로 이동 유도

### 19.4 오류 상태

- API 로드 실패
- 이미지 로드 실패
- 저장 데이터 파싱 실패

## 20. 테스트 전략

### 20.1 기본 기능 테스트

- 첫 화면 렌더링
- 페이지 전환
- 외부 데이터 로드 후 목록 표시
- 검색/필터/정렬
- 카드 선택
- 상세 페이지 전환
- 즐겨찾기 토글
- 설정 반영

### 20.2 시연 포인트 테스트

- 카드 하나를 선택하면 상세 정보가 즉시 반영된다.
- 즐겨찾기 토글 시 대시보드 KPI와 컬렉션이 함께 갱신된다.
- 설정 변경 시 카드 표현 방식이 달라진다.
- 페이지 전환도 결국 루트 상태 변경이라는 점이 검증된다.
- `Render / Patch Inspector`가 이미지 src patch 같은 의미 있는 DOM 갱신을 보여준다.
- Collection은 대량 카탈로그에서도 현재 구간만 렌더되는지 검증한다.

### 20.3 보조 인터랙션 테스트

- 틸트 효과 on/off
- glare 효과 on/off
- 포인터 이탈 시 카드 복귀

이 영역은 상태 테스트보다 DOM 스타일 테스트에 가깝다.

### 20.4 페이지별 테스트 매트릭스

대시보드:

- KPI 카드 렌더링
- 추천 카드 표시
- 최근 액션 반영

컬렉션:

- 검색
- 타입 필터
- 즐겨찾기 필터
- 정렬
- 카드 선택
- 즐겨찾기 토글

상세:

- 선택 카드 반영
- 이전/다음 카드 전환
- 설정에 따른 효과 on/off

설정:

- 기본 시작 페이지 변경
- 기본 정렬 변경
- 틸트/광택 토글
- 리셋 동작

## 21. 페이지별 승인 기준

### 21.1 대시보드 승인 기준

- KPI 카드가 카드 상태와 즉시 동기화된다.
- 추천 카드가 최소 1장 이상 의미 있게 노출된다.
- 최근 액션이 최신 상태를 반영한다.

### 21.2 컬렉션 승인 기준

- 검색, 필터, 정렬, 선택, 즐겨찾기가 모두 동작한다.
- 결과 수가 즉시 바뀐다.
- 선택 상태가 상세 페이지와 연결된다.

### 21.3 상세 승인 기준

- 선택된 카드 정보가 정확히 반영된다.
- 카드 3D 틸트와 광택 효과가 동작한다.
- 효과를 꺼도 카드 기본 렌더는 정상 동작한다.

### 21.4 설정 승인 기준

- 최소 3개 이상의 설정 항목이 동작한다.
- 변경 내용이 다른 페이지에 즉시 반영된다.

## 23. 발표용 핵심 메시지

- 이 앱은 단순 카드 나열이 아니라 실제 서비스처럼 역할이 분리된 상태 기반 다중 페이지 SPA다.
- 루트 상태는 하나지만, 대시보드, 컬렉션, 상세, 설정이 서로 다른 형태로 그 상태를 보여준다.
- 필터, 정렬, 선택, 즐겨찾기, 페이지 전환 모두 v3 라이브러리의 상태와 렌더 파이프라인 위에서 동작한다.
- 카드의 기울기와 광택은 고빈도 DOM 스타일 인터랙션으로 분리해 성능과 시연성을 함께 잡았다.

## 24. 최종 권장 요약

새 시연 앱은 강한 시각적 인상을 줄 수 있다.

이 방향이 적합한 이유는 다음과 같다.

- 청중이 보기에 즉시 재미있다.
- 카드 상태 변화가 화면에 명확하게 보인다.
- 외부 이미지와 실제 UI 효과가 있어 서비스 완성도가 높아 보인다.
- v3 라이브러리의 상태 관리와 렌더링 역할을 더 설득력 있게 보여줄 수 있다.

따라서 이후 구현은 본 문서를 기준으로 `카드 컬렉션 쇼케이스 서비스` 형태로 진행한다.
