## 5. 검증 페이지 활용 가이드

검증 페이지는 발표 자료가 아니라 실시간 설명 도구다.
따라서 각 패널은 다음 역할을 가져야 한다.

발표에서는 테스트 케이스의 존재만 언급하지 말고, 어떤 검증 과정을 거쳤고 어떤 결과를 확인했는지도 함께 보여주는 것이 좋다.

### 5.1 actual panel

- patch 적용 결과를 보여준다.
- 부분 변경이 실제로 일어났다는 점을 강조한다.

### 5.2 test panel

- 사용자가 수정한 목표 상태를 보여준다.
- actual panel과 비교 기준 역할을 한다.

### 5.3 HTML 편집기

- 구조와 속성 변경을 빠르게 실험하게 해준다.
- sanitize 정책 때문에 일부 입력이 제거될 수 있음을 안내한다.

### 5.4 diff log

- 어떤 변경이 감지되었는지 설명한다.
- `auto`와 `index` 모드 차이를 시각적으로 드러내기 좋다.

### 5.5 patch log

- 감지된 변경이 실제 DOM 조작으로 어떻게 이어졌는지 보여준다.

### 5.6 history panel

- 현재 index와 전체 길이를 보여준다.
- undo/redo의 내부 모델 설명에 사용한다.

### 5.7 vnode viewer

- Virtual DOM의 shape를 설명한다.
- DOM과 분리된 중간 표현이라는 점을 보여준다.

## 7. 샘플/시나리오 구성 가이드

### 7.1 초기 샘플에 포함하면 좋은 요소

- 제목과 본문 텍스트
- 중첩된 리스트
- 버튼과 입력 요소
- `data-*` 속성
- class와 style이 섞인 요소

### 7.2 이벤트 시나리오 분리 원칙

초기 HTML 샘플에는 실행형 이벤트를 넣지 않는다.
이벤트 검증은 선언형 시나리오로 별도 준비한다.

권장 예:

- 버튼 클릭 핸들러 변경
- input 핸들러 제거
- 기존 이벤트를 다른 이벤트로 교체
