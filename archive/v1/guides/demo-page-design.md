# React 유사 Virtual DOM / Diff / Patch 시연 페이지 설계 문서

## 1. 문서 목적

이 문서는 시연용 페이지의 화면 구조, 사용자 흐름, 시각적 우선순위, 와이어프레임을 정의한다.

`requirements.md`와 `architecture.md`가 "무엇을 제공해야 하는가"와 "어떻게 계층을 나눌 것인가"를 정한다면, 이 문서는 "사용자와 발표자가 실제 화면에서 무엇을 보게 될 것인가"를 정리한다.

이 문서는 보조 설계 문서이며, 상위 권한 문서와 충돌할 경우 다음 우선순위를 따른다.

1. `architecture.md`
2. `api-spec.md`
3. `requirements.md`
4. 본 문서

## 2. 설계 목표

시연용 페이지는 단순히 버튼을 모아 둔 개발자용 디버그 화면이 아니라, 발표와 이해를 동시에 지원하는 설명형 인터페이스여야 한다.

핵심 목표는 다음과 같다.

1. 한 화면에서 시스템의 전체 흐름을 이해할 수 있어야 한다.
2. Actual DOM, Test DOM, Diff, Patch, History의 역할이 시각적으로 분리되어야 한다.
3. 발표자가 4분 내에 주요 시나리오를 안정적으로 시연할 수 있어야 한다.
4. 사용자가 "무엇을 수정했고 어떤 차이가 감지되었으며 무엇이 실제 DOM에 반영되었는지"를 즉시 이해할 수 있어야 한다.
5. 기본 화면이 너무 밋밋하거나 개발자 도구처럼 보이지 않도록 의도적인 시각 언어를 가져야 한다.

추가로, 이 시연 페이지는 React를 전혀 모르는 사용자도 "지금 무엇을 보고 있고 왜 이런 과정이 필요한가"를 따라갈 수 있어야 한다.
즉 이 페이지는 단순한 조작 화면이 아니라, 짧은 설명을 내장한 학습형 시연 화면이어야 한다.

## 2.1 초심자 친화 목표

초심자 관점에서 반드시 달성해야 할 목표는 다음과 같다.

1. 사용자가 `Patch`, `Undo`, `Redo`를 누르기 전에 각 버튼의 의미를 이해할 수 있어야 한다.
2. "실제 화면", "목표 상태", "가상 DOM", "차이 계산", "부분 반영"의 관계를 쉬운 문장으로 설명해야 한다.
3. 리액트를 몰라도 "전체를 다시 그리는 대신 바뀐 부분만 반영한다"는 핵심 개념을 이해할 수 있어야 한다.
4. 기술 용어는 그대로 두되, 각 용어 옆에 초심자용 한 줄 설명을 함께 제공해야 한다.
5. 발표자가 설명하지 않아도 화면 자체가 최소한의 온보딩 역할을 해야 한다.

## 3. UX 원칙

### 3.1 역할이 다른 패널은 명확히 분리한다

- Actual Panel은 "현재 브라우저에 반영된 결과"를 보여준다.
- Test Panel은 "사용자가 목표 상태로 편집 중인 결과"를 보여준다.
- Editor는 "입력 원본"이다.
- Diff Log는 "감지된 변화"를 보여준다.
- Patch Log는 "실제 DOM에 실행된 작업"을 보여준다.
- History Panel은 "상태 이동 모델"을 보여준다.
- VNode Viewer는 "중간 표현"을 보여준다.

같은 정보를 다른 표현으로 중복해 보여주더라도, 각 패널의 역할은 겹치지 않게 유지한다.

### 3.2 화면은 왼쪽에서 오른쪽으로 흐른다

사용자가 HTML을 수정하고, 비교 결과를 확인하고, 실제 결과를 보는 흐름이 직관적이어야 한다.

권장 순서는 다음과 같다.

1. 입력
2. 목표 상태 미리보기
3. 차이 감지
4. 실제 반영 결과
5. 히스토리 및 내부 표현

### 3.3 발표 중 시선 이동을 최소화한다

- 패널 수는 많더라도 한 번에 설명할 핵심은 2~3개만 눈에 들어와야 한다.
- 로그 패널은 화면 하단에 넓게 두어 보조 정보로 쓰고, 핵심 조작 패널은 상단에 둔다.
- 버튼 그룹은 중앙 상단에 모아 "조작점"이 분산되지 않게 한다.

### 3.4 상태는 색으로, 설명은 텍스트로 보강한다

- 색만으로 의미를 전달하지 않는다.
- 패널 제목, 상태 배지, 로그 태그를 함께 사용한다.
- 위험 동작은 색과 라벨을 함께 사용한다.

### 3.5 화면 안에서 스스로 설명해야 한다

초심자에게는 "이 패널이 왜 필요한가"가 먼저 보이지 않으면 UI가 복잡하게 느껴진다.
따라서 각 핵심 패널에는 짧은 한 줄 설명을 함께 둔다.

예시:

- Actual Panel: `브라우저에 실제로 반영된 현재 결과입니다.`
- Test Panel: `사용자가 목표로 편집 중인 상태입니다.`
- Diff Log: `이전 상태와 비교해 감지한 변화입니다.`
- Patch Log: `감지한 변화를 실제 DOM에 적용한 기록입니다.`
- History Panel: `이전 상태로 되돌아가거나 다시 앞으로 갈 수 있습니다.`

### 3.6 설명은 단계형으로 제공한다

초심자는 모든 개념을 한 번에 이해하지 못한다.
따라서 화면 상단에 현재 시스템의 흐름을 짧은 단계로 나누어 보여주는 것이 좋다.

권장 단계:

1. `Edit Target`
2. `Build VDOM`
3. `Find Diff`
4. `Apply Patch`
5. `Move History`

이 단계는 기술적 정확성뿐 아니라, 사용자가 "지금 어디까지 왔는가"를 알게 해 주는 진행 표시 역할도 한다.

## 4. 정보 구조

시연 페이지는 아래 10개 영역으로 구성한다.

1. 헤더
2. Intro / Onboarding 카드
3. 단계 안내 스텝퍼
4. 컨트롤 바
5. HTML Editor
6. Test Panel
7. Actual Panel
8. Diff / Patch Log
9. History Panel
10. VNode Viewer / Inspect Panel

### 4.1 헤더

헤더에는 아래 요소를 포함한다.

- 프로젝트 제목
- 한 줄 설명
- 현재 diff 모드 배지
- 현재 history index 요약
- 현재 상태 배지

예시 상태 배지:

- `Preview Ready`
- `Diff Generated`
- `Patched`
- `Undo Applied`
- `Redo Applied`
- `Sanitized Input`

### 4.2 컨트롤 바

컨트롤 바는 조작의 중심이며, 다음 요소를 포함한다.

- Scenario 선택 드롭다운
- Diff Mode 선택 세그먼트
  - `Auto`
  - `Index`
  - `Keyed`
- `Patch`
- `Undo`
- `Redo`
- `Reset`
- `Run Tests`

Patch는 가장 강조되는 기본 액션이어야 한다.
Undo / Redo는 History 영역과 함께 동작 의미가 드러나야 한다.
Reset은 위험 액션이므로 강조는 하되 Patch보다 더 강하게 보이지 않게 한다.

### 4.2 Intro / Onboarding 카드

헤더 바로 아래에는 "이 화면이 무엇을 보여주는지"를 짧게 설명하는 온보딩 카드를 둔다.

권장 포함 요소:

- 한 줄 소개
- "어떻게 사용하면 되는가" 3단계 안내
- 핵심 개념 요약

예시 문구:

- `왼쪽에서 목표 HTML을 수정하고, Patch를 누르면 오른쪽 실제 DOM에 바뀐 부분만 반영됩니다.`
- `이 페이지는 React의 핵심 아이디어인 Virtual DOM -> Diff -> Patch 흐름을 눈으로 보여줍니다.`

### 4.3 단계 안내 스텝퍼

온보딩 카드 아래에는 현재 흐름을 시각적으로 설명하는 단계 스텝퍼를 둔다.

권장 단계:

1. Target Edit
2. Virtual DOM Build
3. Difference Detection
4. DOM Patch
5. History Update

이 스텝퍼는 단순 장식이 아니라, 사용자가 현재 시스템의 흐름을 따라가게 만드는 설명형 장치다.

### 4.4 HTML Editor

HTML Editor는 사용자가 목표 상태를 입력하는 영역이다.

포함 요소:

- 섹션 제목
- 짧은 안내 문구
- sanitize 상태 표시
- textarea 또는 code editor
- Apply Preview 버튼 또는 자동 미리보기 안내

에디터의 목적은 "실제 DOM을 직접 바꾸는 것"이 아니라 "비교 대상이 될 목표 상태를 만드는 것"임이 분명해야 한다.

### 4.5 Test Panel

Test Panel은 sanitize 결과와 VDOM 렌더 결과를 보여주는 목표 상태 뷰다.

표시 요소:

- 패널 제목
- 현재 입력이 sanitize 되었는지 여부
- 미리보기 콘텐츠
- 편집 결과 요약 배지

### 4.6 Actual Panel

Actual Panel은 patch가 실제로 반영된 결과를 보여주는 핵심 패널이다.

표시 요소:

- 패널 제목
- 현재 실제 DOM 상태
- 마지막 patch 시점 배지
- 변경 성공/실패 표시

이 패널은 시연 중 가장 자주 보게 되는 영역이므로 시각적으로 가장 안정적이고 읽기 쉬워야 한다.

### 4.7 Diff / Patch Log

Diff Log와 Patch Log는 분리하되, 한 구획 안에서 탭 또는 좌우 분할로 배치할 수 있다.

Diff Log는 다음을 보여준다.

- patch type
- path
- key 또는 index 정보
- 요약 설명

Patch Log는 다음을 보여준다.

- 실제 적용 순서
- 성공/실패 상태
- DOM 조작 요약

### 4.8 History Panel

History Panel은 단순 수치 표시가 아니라 "상태 이동 모델"을 설명하는 UI여야 한다.

표시 요소:

- 현재 index / 전체 길이
- snapshot 목록 또는 step indicator
- 현재 위치 강조
- undo 가능 여부
- redo 가능 여부

### 4.9 VNode Viewer / Inspect Panel

이 패널은 발표자가 "실제 DOM과 분리된 중간 표현"을 설명할 때 사용한다.

표시 요소:

- 현재 VNode tree
- 현재 diff mode
- 마지막 patch 개수
- inspect 정보

기본적으로는 접을 수 있어도 좋지만, 발표용에서는 쉽게 펼칠 수 있어야 한다.

## 5. 화면 레이아웃 제안

### 5.1 데스크톱 기본 레이아웃

권장 레이아웃은 상단 설명 영역 + 메인 3열 + 하단 2행 구조다.

```txt
+----------------------------------------------------------------------------------+
| Header: Title | Project Summary | Diff Mode Badge | History Index | Status Badge |
+----------------------------------------------------------------------------------+
| Intro Card: What this demo does | How to use it | Why Virtual DOM matters        |
+----------------------------------------------------------------------------------+
| Stepper: Edit Target -> Build VDOM -> Find Diff -> Apply Patch -> Move History   |
+----------------------------------------------------------------------------------+
| Controls: Scenario | Auto/Index/Keyed | Patch | Undo | Redo | Reset | Run Tests |
+----------------------------------------------------------------------------------+
| HTML Editor                  | Test Panel                  | Actual Panel        |
| textarea / editor            | sanitized preview           | patched result      |
| helper text                  | target state summary        | actual DOM summary  |
+----------------------------------------------------------------------------------+
| Diff Log                                         | Patch Log                           |
| detected changes                                 | applied DOM operations              |
+----------------------------------------------------------------------------------+
| History Panel                                    | VNode Viewer / Inspect Panel        |
```

- 편집, 목표 상태, 실제 상태가 한 줄에 나란히 배치된다.
- 발표자가 "왼쪽에서 오른쪽"으로 흐름을 설명하기 쉽다.
- 보조 정보는 아래에 모아 시각적 우선순위를 유지할 수 있다.

## 7. 사용자 흐름

## 7.1 초심자 온보딩 흐름

처음 들어온 사용자는 아래 순서로 자연스럽게 이해할 수 있어야 한다.

1. 헤더에서 이 페이지가 "React-like rendering flow"를 보여준다는 사실을 본다.
2. Intro 카드에서 "왼쪽에서 수정하고, 가운데서 목표 상태를 보고, 오른쪽에 실제 결과가 반영된다"는 설명을 읽는다.
3. 단계 스텝퍼를 통해 시스템이 내부적으로 어떤 순서로 움직이는지 본다.
4. 패널 하단의 짧은 설명 문구를 읽으며 각 영역의 역할을 구분한다.
5. Patch를 눌렀을 때 Diff Log와 Patch Log를 함께 보며 "비교"와 "적용"이 다르다는 점을 이해한다.
6. Undo / Redo를 통해 상태 이력 개념을 확인한다.

### 7.2 초기 로드

1. 초기 샘플 HTML을 Actual Panel에 렌더링한다.
2. Actual DOM을 `domToVNode()`로 변환한다.
3. 변환된 VNode를 초기 상태로 저장한다.
4. 같은 VNode를 Test Panel에 렌더링한다.
5. History Panel은 `entries[0]` 상태를 표시한다.
6. VNode Viewer에는 초기 canonical VNode를 보여준다.

### 7.3 HTML 편집

1. 사용자가 Editor에서 HTML을 수정한다.
2. 시스템은 sanitize 여부를 보여준다.
3. Test Panel은 목표 상태 미리보기로 갱신된다.
4. 아직 Actual Panel은 바뀌지 않는다.
5. 상태 배지는 `Preview Ready`가 된다.

### 7.4 Patch 실행

1. 사용자가 `Patch`를 누른다.
2. Test Panel DOM을 VNode로 변환한다.
3. 이전 VNode와 새로운 VNode를 비교한다.
4. Diff Log가 갱신된다.
5. Patch를 Actual Panel에 적용한다.
6. Patch Log가 갱신된다.
7. History Panel이 한 칸 앞으로 이동한다.
8. 상태 배지는 `Patched`가 된다.

### 7.5 Undo / Redo

1. 사용자가 Undo를 누른다.
2. History index가 감소한다.
3. Actual Panel은 선택된 snapshot으로 되돌아간다.
4. Test Panel과 Editor는 demo 계층에서 같은 currentVNode 기준으로 동기화된다.
5. 상태 배지는 `Undo Applied`가 된다.

Redo는 같은 원리로 반대 방향으로 동작한다.

### 7.6 Diff 모드 비교

1. Scenario를 재정렬 중심 예제로 바꾼다.
2. `Auto` 모드에서 patch 결과를 본다.
3. `Index` 모드로 바꿔 다시 비교한다.
4. 필요 시 `Keyed` 모드로도 비교한다.
5. Diff Summary는 모드 차이를 직관적으로 보여줘야 한다.

## 8. 시각 디자인 방향

### 8.1 기본 톤

기본 톤은 "개발자 도구"가 아니라 "설명형 실험실"에 가깝게 잡는다.

권장 방향:

- 배경은 단순 흰색 단색 대신 아주 옅은 온도감이 있는 중성 배경을 사용한다.
- 핵심 패널 카드는 명확한 경계와 충분한 여백을 가진다.
- 색은 기능적 역할에 따라 제한적으로 사용한다.

### 8.2 색 역할

권장 색 역할은 다음과 같다.

- Actual Panel: 안정적인 청록 또는 남색 계열
- Test Panel: 준비 상태를 나타내는 황토 또는 주황 계열
- Editor: 중립적인 회색-베이지 계열
- Diff Log: 변화 감지를 나타내는 코발트 또는 블루 계열
- Patch Log: 실행 결과를 나타내는 녹색 계열
- Reset / 위험 상태: 탁한 적색 계열

색은 배경 전체를 채우기보다 제목 바, 배지, 좌측 보더, 태그에 사용하는 편이 읽기 쉽다.

### 8.3 타이포그래피 방향

- 제목은 기술 데모치고 약간 개성이 있는 세리프 또는 슬랩 계열을 고려할 수 있다.
- 본문과 UI 라벨은 가독성이 좋은 산세리프를 사용한다.
- 코드/로그/패스 표현은 모노스페이스를 사용한다.

중요한 것은 "전부 같은 폰트처럼 보이는 평면적인 화면"을 피하는 것이다.

### 8.4 상태 표현 규칙

- 배지는 짧고 일관된 용어를 쓴다.
- 성공 상태는 녹색, 준비 상태는 파랑, 경고는 주황, 오류는 빨강을 기본으로 한다.
- disabled 버튼은 단순히 흐리게만 하지 말고 이유를 툴팁 또는 보조 문구로 제공한다.

### 8.5 로그 시각화 규칙

- 로그는 최신 항목이 위로 오도록 한다.
- patch type마다 색 태그를 붙인다.
- path, key, index는 모노스페이스로 표기한다.
- 설명 텍스트는 한 줄 요약을 우선한다.

예시:

- `MOVE_CHILD` `path:[0]` `key:c` list reorder detected
- `SET_PROP` `path:[0,1]` `className` updated

## 9. 초심자 설명 UX 설계

### 9.1 용어는 유지하되 쉬운 문장을 함께 둔다

기술 데모의 목적상 `Virtual DOM`, `Diff`, `Patch`, `History`라는 용어는 숨기지 않는다.
대신 모든 용어 옆에 쉬운 한 줄 설명을 병기한다.

예시:

- `Virtual DOM`: `브라우저 화면을 직접 바꾸기 전에 비교하기 위한 중간 트리입니다.`
- `Diff`: `이전 상태와 지금 목표 상태의 차이를 찾는 단계입니다.`
- `Patch`: `찾아낸 차이만 실제 화면에 반영하는 단계입니다.`
- `History`: `이전 상태로 돌아가거나 다시 앞으로 가기 위한 기록입니다.`

### 9.2 패널 제목 아래에 한 줄 설명을 둔다

각 패널의 제목 아래에는 짧고 쉬운 문장이 있어야 한다.

권장 문구:

- HTML Editor: `바꾸고 싶은 목표 HTML을 입력합니다.`
- Test Panel: `입력한 목표 상태를 미리 보여줍니다.`
- Actual Panel: `브라우저에 실제로 반영된 결과입니다.`
- Diff Log: `무엇이 달라졌는지 계산한 결과입니다.`
- Patch Log: `그 차이를 실제 DOM에 적용한 기록입니다.`
- History Panel: `이전 상태와 다음 상태로 이동할 수 있습니다.`
- VNode Viewer: `브라우저 DOM과 분리된 내부 표현입니다.`

### 9.3 버튼에도 설명 문구를 보강한다

버튼 이름만으로 부족한 경우 짧은 보조 설명을 제공한다.

예시:

- `Patch`: `변경된 부분만 실제 화면에 반영`
- `Undo`: `이전 상태로 되돌리기`
- `Redo`: `다음 상태로 다시 이동`
- `Reset`: `초기 상태로 전체 복원`
- `Run Tests`: `핵심 동작 검증`

### 9.4 상태 메시지는 기술 중심보다 사용자 중심으로 쓴다

너무 기술적인 메시지만 보여주면 초심자는 상황을 해석하기 어렵다.

권장 방향:

- `Diff Generated` 보다 `차이를 찾았습니다`
- `Patch Applied` 보다 `실제 화면에 반영되었습니다`
- `Sanitized Input` 보다 `위험한 HTML을 제거하고 안전한 형태로 미리보기를 만들었습니다`

기술적 상태명은 배지에 유지해도 좋지만, 본문 설명은 쉬운 문장을 우선한다.

### 9.5 "왜 필요한가"를 한 번 더 설명한다

초심자는 "왜 굳이 중간 표현을 만들고 비교하는가"에서 가장 많이 막힌다.
따라서 Intro 카드나 도움말 영역에 아래 요지를 항상 드러내는 것이 좋다.

- `브라우저 DOM을 바로 계속 바꾸는 대신, 먼저 가상 트리에서 차이를 계산하면 더 통제된 방식으로 변경할 수 있습니다.`
- `이 데모는 리액트가 내부적으로 하는 핵심 아이디어를 단순화해서 보여줍니다.`

## 10. 발표용 문구 예시

### 10.1 비전공자 / 초심자용 설명

- `왼쪽은 바꾸고 싶은 목표 상태입니다.`
- `가운데는 그 목표 상태를 미리 보여주는 영역입니다.`
- `오른쪽은 브라우저에 실제로 반영된 현재 상태입니다.`
- `Patch를 누르면 시스템이 둘의 차이를 계산해서, 바뀐 부분만 실제 화면에 반영합니다.`
- `Undo와 Redo는 그 변경 기록을 왔다 갔다 보여줍니다.`

### 10.2 기술 설명용 문장

- `이 시스템은 먼저 DOM을 VNode라는 중간 트리로 표현합니다.`
- `그 다음 이전 VNode와 새로운 VNode를 비교해 flat patch list를 만듭니다.`
- `Patch 단계는 그 리스트를 순서대로 실행하면서 실제 DOM을 부분 갱신합니다.`
- `History는 entries + currentIndex 모델로 관리합니다.`

### 10.3 리액트와 연결하는 설명

- `리액트 전체를 그대로 구현한 것은 아니지만, 가상 트리 비교 후 필요한 변경만 반영한다는 핵심 흐름은 비슷합니다.`
- `특히 key-aware diff 모드를 통해 리스트 재정렬을 더 자연스럽게 다루는 방향을 보여줍니다.`

## 11. 발표용 UX 주의사항

1. 첫 화면에서 너무 많은 로그가 보이지 않게 한다.
2. Actual Panel과 Test Panel이 시각적으로 뒤바뀌어 보이면 안 된다.
3. Patch 버튼은 눈에 잘 띄어야 하지만 화면 전체를 압도하면 안 된다.
4. Undo / Redo의 불가능 상태는 비활성으로 분명히 보인다.
5. Reset은 실수 클릭을 줄이기 위해 다른 조작 버튼과 약간 거리를 둔다.
6. 테스트 실행 결과는 시연 흐름을 끊지 않도록 접거나 별도 영역에 둔다.
7. VNode Viewer는 고급 정보이므로 처음부터 화면의 주인공이 되지 않게 한다.

## 12. 구현 체크리스트

### 12.1 반드시 만족해야 할 항목

- Actual Panel, Test Panel, Editor, Diff Log, Patch Log, History, VNode Viewer가 모두 존재한다.
- Intro 카드와 단계 스텝퍼가 존재한다.
- `Patch`, `Undo`, `Redo`, `Reset`, `Run Tests`가 모두 눈에 띄게 배치된다.
- `Auto`, `Index`, `Keyed` 모드를 전환할 수 있다.
- 초기 로드 흐름이 요구사항 문서와 일치한다.
- Undo / Redo 이후 Test Panel과 Editor가 함께 동기화된다.
- Diff Log와 Patch Log의 역할이 분리된다.
- 각 핵심 패널에 초심자용 한 줄 설명이 들어간다.
- 주요 상태 메시지가 기술자용 용어만으로 끝나지 않는다.
- 모바일에서도 레이아웃이 깨지지 않는다.

### 12.2 품질을 높이는 항목

- 시연용 scenario를 빠르게 바꿀 수 있다.
- `Auto`와 `Index` 비교 결과를 별도 요약 카드로 보여준다.
- sanitize 결과를 배지나 안내 문구로 명확히 표시한다.
- patch success / failure 메시지가 명확하다.
- 로그가 너무 길어질 경우 필터 또는 접기 기능이 있다.

## 13. 비목표

다음은 시연용 페이지의 1차 비목표다.

- 범용 IDE 수준의 코드 편집기 제공
- 디자이너용 고급 테마 시스템 제공
- 실시간 협업 UI
- 복잡한 애니메이션 중심 랜딩 페이지 스타일
- DevTools 수준의 트리 탐색기 완전 구현

## 14. 최종 권장 방향

현재 프로젝트의 목적을 고려하면, 시연 페이지는 "예쁜 대시보드"보다 "설명력이 높은 실험실"에 가까워야 한다.

즉 다음 세 가지를 동시에 만족하는 방향이 가장 적합하다.

1. Actual / Test / Diff / Patch / History의 관계가 한눈에 보인다.
2. 발표자가 왼쪽에서 오른쪽, 위에서 아래 순서로 흐름을 설명할 수 있다.
3. 기본적인 시각 완성도가 있어 초라한 개발자 테스트 화면처럼 보이지 않는다.

이 문서를 기준으로 실제 UI를 구현할 때는 먼저 데스크톱 레이아웃과 핵심 상태 표현을 완성하고, 그 다음 모바일 대응과 시각 디테일을 다듬는 순서를 권장한다.
