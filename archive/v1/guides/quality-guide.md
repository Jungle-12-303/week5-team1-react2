# React 유사 Virtual DOM / Diff / Patch 시스템 품질 가이드

## 1. 문서 목적

본 문서는 결과물 수준, 코드 품질, 테스트 품질, 검증 가능성에 대한 상세 기준을 정리한다.
이 문서는 `requirements.md`의 품질 요구사항을 보조적으로 구체화한다.

본 문서는 보조 문서다.
상위 권한 문서와 충돌할 경우 상위 권한 문서를 따른다.

## 2. 결과물 수준 기준

결과물은 아래 다섯 관점에서 균형 있게 완성되어야 한다.

### 2.1 구현 완성도

- 핵심 기능이 실제로 작동해야 한다.
- patch가 변경된 부분만 반영해야 한다.
- undo / redo가 안정적으로 동작해야 한다.
- 테스트 페이지에서 다양한 변경을 시연할 수 있어야 한다.

### 2.2 설명 가능성

- 발표자가 핵심 알고리즘 흐름을 설명할 수 있어야 한다.
- 코드가 과도하게 축약되어 있으면 안 된다.
- 모듈별 책임이 드러나야 한다.
- 핵심 분기의 이유를 README와 코드 주석으로 설명할 수 있어야 한다.

### 2.3 검증 가능성

- 테스트 케이스가 포함되어야 한다.
- patch 결과를 눈으로 확인할 수 있어야 한다.
- diff 결과를 로그나 시각화로 확인할 수 있어야 한다.
- 실패 케이스를 진단할 수 있는 정보가 있어야 한다.

### 2.4 재사용 가능성

- 외부 프로젝트에서 라이브러리처럼 사용할 수 있어야 한다.
- 공개 API가 내부 구현 세부에 과하게 묶이지 않아야 한다.
- 최소한의 사용 예제가 존재해야 한다.

### 2.5 확장 가능성

- component/state/runtime 관련 모듈을 나중에 추가할 수 있어야 한다.
- renderer abstraction을 도입할 여지가 있어야 한다.
- 현재 구조가 future React-like architecture로 자연스럽게 성장 가능해야 한다.

## 3. 코드 품질 기준

### 3.1 코드 작성 원칙

1. 의미 있는 함수명 사용
2. 축약 남용 금지
3. 하나의 함수는 하나의 책임 중심
4. 함수 길이는 가능한 한 짧게 유지
5. 핵심 로직을 숨기는 과도한 추상화 금지
6. 알고리즘을 설명 가능한 단계로 분리

### 3.2 가독성 기준

- minified style 코드 금지
- one-liner 위주 난독화 금지
- 설명 없이 복잡한 조건문 작성 금지
- 이유 없는 재귀/고차함수 남용 금지
- 핵심 로직의 복붙식 누적 금지

### 3.3 주석 기준

모듈 상단에는 다음 정보가 있으면 좋다.

- 파일의 책임
- 입력/출력
- 다른 모듈과의 관계
- 향후 확장 포인트

핵심 함수에는 다음 정보가 있으면 좋다.

- 함수 목적
- 입력
- 반환값
- 부작용
- 알고리즘 요약

내부 주석이 특히 필요한 위치:

- auto/index/keyed 분기
- DOM 재사용 여부 판단
- 이벤트 교체 로직
- history 잘라내기 로직
- 향후 component/runtime 확장 고려 지점

## 4. 테스트 품질 기준

### 4.1 필수 테스트 분류

- VNode 생성/정규화 테스트
- DOM -> VDOM 테스트
- VDOM -> DOM 테스트
- diff 테스트
- patch 테스트
- history 테스트
- engine 통합 테스트
- demo와 core 분리 검증 테스트

### 4.2 핵심 테스트 케이스

최소 다음 케이스가 포함되어야 한다.

- text 변경
- class 변경
- style 변경
- 속성 추가
- 속성 제거
- child 추가
- child 삭제
- 태그 변경
- node replace
- key 기반 재정렬
- explicit keyed 모드 비교
- 이벤트 변경
- property 기반 입력값 반영
- undo 후 redo 제거

### 4.3 테스트 결과 표현

테스트 결과는 최소 다음 정보를 전달해야 한다.

- pass / fail 개수
- 실패한 케이스 이름
- 기대값 / 실제값 요약
- 필요 시 실패 시점의 patch 또는 vnode 일부

### 4.4 구조 검증 테스트

다음 구조적 조건도 테스트 또는 점검 대상으로 포함하는 것이 좋다.

- core가 demo 없이 import 가능해야 한다.
- renderer-dom 없이 diff 단위 테스트가 가능해야 한다.
- history가 UI 코드 없이 독립 테스트 가능해야 한다.
- 공개 API가 내부 파일 경로에 의존하지 않아야 한다.

## 5. 검증 페이지 품질 기준

검증 페이지는 단순한 버튼 모음이 아니라 발표용 진단 도구 역할을 해야 한다.

필수 확인 항목:

- 실제 영역과 테스트 영역의 차이가 육안으로 보이는가
- patch 전후 비교가 가능한가
- 현재 history index를 이해하기 쉬운가
- diff 결과를 사람이 읽을 수 있는가
- patch log가 실제 동작과 연결되어 보이는가
- 현재 vnode를 inspect 할 수 있는가

권장 항목:

- patch 적용 시간 표시
- 전체 재렌더링과 부분 patch 비교
- key 모드 토글
- 변경된 노드 수 표시

## 6. 샘플 시나리오 품질 기준

초기 샘플은 다양한 태그와 요소를 포함하는 것이 좋다.

권장 포함 요소:

- heading
- paragraph
- list
- button
- input
- nested container
- data attribute
- boolean attribute

샘플이 지원해야 하는 변화:

- 텍스트 변경
- class 변경
- style 변경
- 속성 추가/삭제
- child 추가/삭제
- child 재정렬
- 태그 변경
- 이벤트 변경
- 입력값 변화

## 7. 엣지 케이스 품질 기준

다음은 처리 또는 문서화 수준을 반드시 확보해야 한다.

### 7.1 구조 관련

- 빈 children
- null children
- 연속 text node
- whitespace-only text node
- 브라우저가 자동 보정한 DOM 구조

### 7.2 diff 관련

- oldVNode만 존재
- newVNode만 존재
- 태그/type 완전 변경
- key 충돌
- 동일 key 중복

### 7.3 patch 관련

- 대상 path 없음
- 삭제 후 인덱스 밀림
- 이벤트 중복 연결
- prop 제거 후 property 잔존
- 이동과 삭제가 동시에 발생하는 경우

### 7.4 history 관련

- 첫 상태에서 undo
- 마지막 상태에서 redo
- 빈 history
- 최대 길이 초과
- currentIndex 범위 이탈

### 7.5 사용자 입력 관련

- 잘못된 HTML 입력
- script 태그 입력
- 이벤트 속성 인라인 입력
- 대량 노드 붙여넣기

## 8. 성능 관점의 품질 기준

이번 과제는 React 수준의 극단적 최적화를 목표로 하지는 않는다.
그러나 다음 기준은 만족하는 것이 좋다.

- 소규모~중간 규모 DOM 트리에서 즉각 반응
- patch 버튼 클릭 후 체감상 큰 지연 없음
- domToVNode, diff, patch 시간을 측정 가능
- 변경된 노드 수를 확인 가능

## 9. 포트폴리오 관점의 품질 기준

포트폴리오 수준의 결과물로 보이려면 다음 요소가 필요하다.

- 구조가 설명 가능하다.
- 단순 `innerHTML` 덮어쓰기가 아니라는 점이 분명하다.
- 테스트와 검증 흔적이 있다.
- README만 보고도 프로젝트의 가치가 전달된다.
- 현재 한계와 향후 확장 계획을 솔직하게 설명한다.
